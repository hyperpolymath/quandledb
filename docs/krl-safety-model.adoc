= KRL Safety Model — Two-Tier Architecture
:toc: left
:toclevels: 3
:sectnums:

== Overview

KRL (Knot Resolution Language) follows the same two-tier model as VCL:

[cols="1,2,3", options="header"]
|===
| Tier | What | How

| **KRL**
| The resolution language
| Pipeline syntax (`from knots \| where ... \| return ...`), parsed and executed.
  What users write.

| **KRL Safety Pipeline**
| Progressive type safety
| Applies TypeLL's 10 levels behind KRL automatically. Simple queries exit early.
  Proof-carrying queries (with `PROOF` clause) activate L9-L10.
|===

== Relationship to VCL-UT

The 10 safety levels originate in **TypeLL** (the core type theory). VCL-UT is
the reference implementation for database queries. KRL maps its own constructs
to the same 10 levels — it does not have a separate "-UT" repo.

[source]
----
TypeLL (type theory core — defines the 10 levels)
   │
   ├──→ VCL-UT  (reference implementation, VeriSimDB)
   ├──→ KRL     (applies same levels, QuandleDB)
   └──→ GQL     (applies same levels, LithoGlyph)
----

== KRL Safety Levels

[cols="^1,<3,<5", options="header"]
|===
| Level | Name | KRL Mapping

| L1  | Construction Safety    | Pipeline injection prevention (safe `from`/`where`/`return` parsing)
| L2  | Schema Pinning         | Knot record schema checked against database (dependent projection)
| L3  | Resource Linearity     | Connection/cursor leak prevention for Skein.jl backend
| L4  | Session Protocols      | Query session state machine (connect → query → close)
| L5  | Effect Tracking        | Side-effect auditing for mutations (future: write support)
| L6  | Scope Isolation        | Namespace boundaries between knot tables
| L7  | Information Flow       | Knot data lineage (provenance of computational invariants)
| L8  | Quantitative Bounds    | Bounded evaluation (stratum-bounded, proving termination)
| L9  | Proof Attachment       | Confidence-indexed results with Idris2 proofs
| L10 | Cross-Cutting          | Composition safety across pipeline stages
|===

== KRL-Specific Type Features

KRL has four dependent type features that map naturally to the safety levels:

[cols="<3,^1,<4", options="header"]
|===
| Feature | Level | Description

| Dependent Projection (pipeline type refinement)
| L2, L10
| Return type computed from field list. `Π(τ, fields) = τ'`. Idris2 proof: `Project` witnesses.

| Confidence-Indexed Results
| L9
| Equivalence results carry confidence level as part of the type.
  `ResultSet[ConfidenceIdx n τ]` where `n` is the confidence bound.

| Invariant-Dependent Records
| L9
| Available fields depend on which invariants were requested.
  Requesting `jones_polynomial` changes the result type.

| Stratum-Bounded Evaluation
| L8
| Evaluation strategy indexed by invariant strata count.
  Proves termination via decreasing stratum index.
|===

== Historical Note: KRL-DT

The dependent type features documented in `spec/dependent-type-variants.md` were
originally called "KRL-DT". This is not a separate product or execution mode — it is
the description of how KRL's type system uses dependent types, now folded into the
safety pipeline at levels L2, L8, L9, and L10.

All references to "KRL-DT" should be read as "KRL's dependent type features within
the safety pipeline".
